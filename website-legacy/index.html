Colour Palette

Codacy is committed to complying with AA standard contrast ratios. To do this, we use primary and extended colors that support usability by ensuring sufficient color contrast between elements.

Neutral

Neutrals are typically used for text, backgrounds and secondary elements. Dark neutrals are primarily used for copy. Mid-neutrals to create depth in components such as modals and cards. The mid-neutrals can also be used for subtle text and icons. Light neutrals are primarily used as background colors when we want to draw some attention to a particular section or convey information that is “disabled”.

$brand-neutral-900 is our default text color, used in headlines and body.

$brand-neutral-800 this is a color shade variante.

$brand-neutral-700 this is a color shade variante.

$brand-neutral-600 is used for complementary text.

$brand-neutral-500 is used in icons under lighter shades of gray.

$brand-neutral-400 is our default icon color on white background.

$brand-neutral-300 is used as text color on darker backgrounds. Ie - Navbar dropdown.

$brand-neutral-200 is used for lines and borders to create contrast on lighter backgrounds.

$brand-neutral-100 is used as color background in active states. Ie - Button radio, dropdowns. It is also used on hover states, within dropdowns.

$brand-neutral-50 is used as color background for contrasting areas. Ie - Tables, side menu, panels.

$brand-neutral-0 is used as the primary color of the interface.

Blue

Blue is used to indicate authentication, active states, or progress. You’ll find blue in messages, buttons, navigation, drop-downs, tabs, popovers, tooltips and info states.

$brand-primary is used for buttons, links, inputs on focus.

$brand-success is used for successful status and feedback.

$brand-warning is used for warning status.

$brand-danger is used for error status and feedback.

Typography

The main typeface is Clear sans. It should be used on every text in the application. If, for some reason, Clear sans is not rendered, the alternate typeface is Arial.

Buttons

Buttons are to be used whenever there's an action a user can make, therefore, there should be one primary button per page.
.btn-default and .btn-link are the only buttons that can repeatedly appear in a page and should not be used for primary or breaking actions. Buttons should be positioned on the left. When there's a group of buttons, the primary button stands on the far right.

Buttons and tooltips

You can always add a tooltip associated to your button, unless that button has a disabled state. In this case you will need to add a wrapper that handles the tooltip, as the disabled state of the button ignores any action to it, even hover.

Forms

Inputs can be used in a form group or individually. It's mandatory that each input has a label. Inputs should have the appropriate width according to the content it's going to receive - to size the width of an input add the class .col-*-* to .form-group, and to size the height use .input-* or .form-group-* when using .form-groups.

Optional fields should be indicated instead of required fields. Placeholders should always give some hint to the user on what is the expected input.
In case of inputs with feedback, the input should return to it's normal state when it becomes active.

Why is this an issue?

                                

                                  
The use of a predictable random value can lead to vulnerabilities when used in certain security critical contexts. For example, when the value is used as:

                                  
    
                                    
  • a CSRF token
    • 
                                    
  • a password reset token (sent by email)
    • 
                                    
  • any other secret value
    • 
                                    
  • A quick fix could be to replace the use of java.util.Random with something stronger, such as java.security.SecureRandom.
    • 
                                  

                                  
Vulnerable Code:

                                  
String generateSecretToken() {
Random r = new Random();
return Long.toHexString(r.nextLong());
}



                                  
Solution:

                                  

import org.apache.commons.codec.binary.Hex;

String generateSecretToken() {
SecureRandom secRandom = new SecureRandom();

byte[] result = new byte[32];
secRandom.nextBytes(result);
return Hex.encodeHexString(result);
}
